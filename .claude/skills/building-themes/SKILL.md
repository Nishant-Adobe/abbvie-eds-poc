# Building Themes

This skill guides you through managing the multi-brand and multi-theme system in the EDS Multi-Brand project. The project supports multiple brands (e.g. `roy`) and multiple themes per brand (e.g. `bright`). Each combination has its own compiled CSS that layers global base styles → brand overrides → theme overrides.

## Related Skills

- **building-blocks**: Use when a new block needs to be created as part of a theme or brand implementation
- **building-brand-blocks**: Use when adding block-level brand/theme CSS overrides alongside a new or existing block
- **content-driven-development**: Use when brand/theme work is driven by a new content requirement

## When to Use This Skill

Use this skill when:
- Adding a **new brand** to the project
- Adding a **new theme** (cross-brand or brand-specific)
- Customising **design tokens** (colors, fonts, spacing) for a brand or theme
- Customising **global styles** (typography, layout, sections) for a brand or theme
- Removing a brand or theme
- Troubleshooting CSS build issues

## Key Concepts

### Brands vs Themes

- **Brand**: A distinct site identity (e.g. `roy`). Has its own set of design tokens, global styles, and block overrides. Corresponds to a sub-path (e.g. `/roy/...`) in the AEM project.
- **Theme**: A visual variant applied within a brand (e.g. `bright`). Layered on top of the brand styles. Applied via a class or metadata on the page.

Both are registered in `brand-config.json`:

```json
{
  "brands": ["roy"],
  "themes": ["bright"]
}
```

### CSS Cascade

Styles apply in this order (later overrides earlier):

```
styles/tokens.css                       ← global design tokens (:root CSS variables)
styles/styles.css                       ← global base styles
styles/fonts.css                        ← global fonts
styles/lazy-styles.css                  ← global lazy-loaded styles

styles/{brand}/_tokens.css             ← brand token overrides
styles/{brand}/_styles.css             ← brand style overrides
styles/{brand}/_fonts.css              ← brand font overrides
styles/{brand}/_lazy-styles.css        ← brand lazy style overrides

styles/{brand}/themes/{theme}/_tokens.css     ← brand+theme token overrides
styles/{brand}/themes/{theme}/_styles.css     ← brand+theme style overrides
styles/{brand}/themes/{theme}/_fonts.css      ← brand+theme font overrides
styles/{brand}/themes/{theme}/_lazy-styles.css

styles/themes/{theme}/_tokens.css      ← cross-brand theme token overrides
styles/themes/{theme}/_styles.css      ← cross-brand theme style overrides
```

**Same underscore rule as blocks:** `_file.css` are the **source partials** (edit these). `file.css` files are **compiled outputs** (generated by `npm run scaffold:build`). Never manually edit compiled files.

### Compiled vs Source Files

| File | Purpose |
|---|---|
| `_tokens.css` | Source: define/override `:root` CSS variables |
| `_styles.css` | Source: override global typography, layout, section styles |
| `_fonts.css` | Source: override or add `@font-face` declarations |
| `_lazy-styles.css` | Source: styles loaded lazily (non-critical, after LCP) |
| `tokens.css` | Compiled output — do not edit |
| `styles.css` | Compiled output — do not edit |
| `fonts.css` | Compiled output — do not edit |
| `lazy-styles.css` | Compiled output — do not edit |

### `styles/` Directory Structure

```
styles/
├── tokens.css               # Global tokens (base)
├── styles.css               # Global base styles
├── fonts.css                # Global fonts
├── lazy-styles.css          # Global lazy styles
├── themes/
│   └── {theme}/             # Cross-brand theme overrides
│       ├── _tokens.css
│       ├── _styles.css
│       ├── _fonts.css
│       ├── _lazy-styles.css
│       └── *.css (compiled)
└── {brand}/
    ├── _tokens.css          # Brand token overrides
    ├── _styles.css          # Brand style overrides
    ├── _fonts.css
    ├── _lazy-styles.css
    ├── *.css (compiled)
    └── themes/
        └── {theme}/         # Brand + theme overrides
            ├── _tokens.css
            ├── _styles.css
            ├── _fonts.css
            ├── _lazy-styles.css
            └── *.css (compiled)
```

### No Hardcoded Values

> **STRICT RULE — No hardcoded values, ever.**
> All token values and CSS property values MUST reference a design token via `var(--token-name)`. Raw values (`#fff`, `1rem`, `600px`, `bold`, `16px`, inline font stacks, etc.) are **never** allowed. The only permitted exception is `0` (the unitless zero).
>
> **If a token does not exist yet, define it first at the correct level of the multi-brand hierarchy — then reference it:**
>
> | Scope | Define the token in |
> |---|---|
> | All brands and themes | `styles/tokens.css` — global base scale |
> | One brand, all themes | `styles/{brand}/_tokens.css` |
> | All brands, one theme | `styles/themes/{theme}/_tokens.css` |
> | One brand + one theme | `styles/{brand}/themes/{theme}/_tokens.css` |
>
> Token values must themselves reference another token via `var(--...)` — never a raw value. Read the existing token files at execution time to find the right reference. Follow W3C DTCG naming: `--{type}-{semantic}-{variant}` (e.g. `--color-brand-surface`, `--spacing-section-gap`).

This rule applies at every level: global, brand, theme, and brand+theme — both in **token definitions** (what a `--variable` resolves to) and **CSS property values** (what you assign to `background-color`, `padding`, etc.).

## Process: Adding a New Brand

### 1. Run the scaffold command

```sh
npm run scaffold:create
# Choose: Brand → enter brand name (lowercase, kebab-case)
```

This runs `node theme-tools/initiate-brand.js brand {name}` which:
- Creates `styles/{brand}/` with all `_*.css` source partials
- Adds the brand to `brand-config.json`
- Creates brand-level `_*.css` folders for all existing blocks in `blocks/`
- Creates brand+theme folders for all existing themes in `brand-config.json`

### 2. Configure brand tokens

Edit `styles/{brand}/_tokens.css` to define brand-specific design tokens. Read `styles/tokens.css` at execution time for the current token names — the structure below is illustrative only; actual token names and values must come from that file. Every value **must** reference another token via `var(--…)` — no raw hex, px, or font-stack literals.

```css
/* styles/{brand}/_tokens.css — illustrative structure; values must reference base tokens */
:root {
  /* Override global base tokens — names must match styles/tokens.css exactly */
  --background-color: var(--color-base-neutral-000);
  --light-color: var(--color-base-neutral-100);
  --dark-color: var(--color-base-neutral-800);
  --text-color: var(--color-base-neutral-900);
  --link-color: var(--color-brand-primary);
  --link-hover-color: var(--color-brand-primary-dark);

  /* Brand fonts — reference font-family tokens, never hardcode stacks */
  --body-font-family: var(--font-family-brand-sans);
  --heading-font-family: var(--font-family-brand-display);

  /* Nav */
  --nav-height: var(--spacing-nav-height);

  /* New brand-specific tokens — W3C DTCG naming: --{type}-{semantic}-{variant} */
  --color-brand-primary: var(--color-scale-blue-600);
  --color-brand-secondary: var(--color-scale-orange-500);
}
```

All tokens defined in `styles/tokens.css` can be overridden here. New brand-specific tokens must follow W3C DTCG naming (`--{type}-{semantic}-{variant}`) and must reference an existing base-scale token via `var(--…)`, never a raw value.

### 3. Add brand fonts (if needed)

Edit `styles/{brand}/_fonts.css`:

```css
/* styles/{brand}/_fonts.css */
@import '../fonts.css';

@font-face {
  font-family: 'Brand Font';
  font-weight: 400;
  font-style: normal;
  font-display: swap;
  src: url('/styles/{brand}/fonts/brand-font-regular.woff2') format('woff2');
}
```

### 4. Override global styles (if needed)

Edit `styles/{brand}/_styles.css`. All property values must reference a token via `var(--…)` — no raw literals or color keywords.

```css
/* styles/{brand}/_styles.css — illustrative structure; values must reference tokens */
@import '../styles.css';

/* Brand-specific typography overrides */
h1 {
  letter-spacing: var(--letter-spacing-heading);
}

/* Brand-specific section overrides */
main > .section.hero {
  background-color: var(--color-brand-primary);
  color: var(--color-text-inverse);
}
```

### 5. Build and verify

```sh
npm run scaffold:build   # Compile CSS
npm start                # Start dev server and verify in browser
```

### 6. Add block-level brand overrides (if needed)

For each block needing brand customisation, edit `blocks/{block-name}/{brand}/_{block-name}.css`. See the **building-brand-blocks** skill for block-level CSS patterns.

## Process: Adding a New Theme

### 1. Run the scaffold command

```sh
npm run scaffold:create
# Choose: Theme → enter theme name (lowercase, kebab-case)
```

This runs `node theme-tools/initiate-brand.js theme {name}` which:
- Creates `styles/themes/{theme}/` with all `_*.css` source partials
- Adds the theme to `brand-config.json`
- Creates `styles/{brand}/themes/{theme}/` for each existing brand
- Creates `blocks/{block-name}/themes/{theme}/` for all existing blocks

### 2. Configure theme tokens

Themes typically override a subset of tokens. Edit `styles/themes/{theme}/_tokens.css`. Every value **must** reference a base token via `var(--…)` — no raw hex, px, or color keyword literals.

```css
/* styles/themes/{theme}/_tokens.css — illustrative structure; values must reference base tokens */
:root {
  /* Theme overrides — only what changes from the brand defaults */
  --background-color: var(--color-base-neutral-950);
  --light-color: var(--color-base-neutral-900);
  --text-color: var(--color-base-neutral-100);
  --link-color: var(--color-brand-primary-light);
  --link-hover-color: var(--color-brand-primary-lighter);
}
```

For a brand-specific theme variant, edit `styles/{brand}/themes/{theme}/_tokens.css`:

```css
/* styles/{brand}/themes/{theme}/_tokens.css — illustrative; values must reference base tokens */
:root {
  /* Overrides for brand + theme combination */
  --background-color: var(--color-base-yellow-100);
}
```

**W3C Design Token naming rules (apply at every level — global, brand, theme, brand+theme):**

All new tokens must follow the [W3C Design Token Community Group](https://www.w3.org/community/design-tokens/) format — type prefix first, then semantic group, then variant:

```
--{type}-{semantic-group}-{variant}
```

```css
/* CORRECT — type prefix first, semantic and portable; values reference base tokens, never hardcoded */
--color-background-default: var(--color-scale-neutral-950);
--color-background-subtle: var(--color-scale-neutral-900);
--color-text-primary: var(--color-scale-neutral-100);
--color-text-link: var(--color-scale-blue-400);
--color-brand-primary: var(--color-scale-blue-400);
--color-brand-surface: var(--color-scale-neutral-900);

/* WRONG — no type prefix */
--brand-primary: var(--color-scale-blue-400);
--brand-surface: var(--color-scale-neutral-900);

/* WRONG — theme name embedded in the token name */
--bright-background: var(--color-scale-yellow-050);
--dark-text-color: var(--color-scale-neutral-100);

/* WRONG — brand name embedded */
--roy-primary: var(--color-scale-blue-600);
```

| Type prefix | Used for |
|---|---|
| `--color-` | All color values |
| `--font-family-` | Font stack declarations |
| `--font-size-` | Text size values |
| `--font-weight-` | Font weight values |
| `--spacing-` | Margin, padding, gap |
| `--border-` | Border width, radius, style |
| `--shadow-` | Box or text shadows |
| `--motion-` | Duration, easing |

**Note on legacy global tokens:** The project's existing base tokens (`--background-color`, `--text-color`, etc. in `styles/tokens.css`) use non-DTCG legacy names. When overriding these in brand or theme `_tokens.css` files, match the existing name exactly. Only apply DTCG naming to **new** tokens you introduce.

### 3. Override theme styles (if needed)

Edit `styles/themes/{theme}/_styles.css`. All property values must reference a token via `var(--…)` — no raw literals.

```css
/* styles/themes/{theme}/_styles.css — illustrative; values must reference tokens */
@import '../../styles.css';

/* Theme-specific style overrides */
main > .section {
  border-radius: var(--border-radius-section);
}
```

### 4. Build and verify

```sh
npm run scaffold:build
npm start
```

### 5. Add block-level theme overrides (if needed)

Edit `blocks/{block-name}/themes/{theme}/_{block-name}.css`. See the **building-brand-blocks** skill.

## Process: Removing a Brand or Theme

```sh
npm run scaffold:remove
# Choose: Brand or Theme → enter name → confirm
```

This removes:
- The `styles/{brand}/` or `styles/themes/{theme}/` directory
- Brand/theme subfolders from all blocks
- The entry in `brand-config.json`

**Warning:** This is irreversible. Ensure the brand/theme is no longer in use before removing.

## Process: Customising Tokens and Styles

### Editing design tokens

1. Identify the token to override in `styles/tokens.css` (the global source of truth)
2. Add the override in the appropriate `_tokens.css` file:
   - Brand-only: `styles/{brand}/_tokens.css`
   - Theme-only (cross-brand): `styles/themes/{theme}/_tokens.css`
   - Brand+Theme: `styles/{brand}/themes/{theme}/_tokens.css`
3. Run `npm run scaffold:build` to recompile

### Available base tokens (from `styles/tokens.css`)

```css
:root {
  /* Colors */
  --background-color, --light-color, --dark-color
  --text-color, --link-color, --link-hover-color

  /* Fonts */
  --body-font-family, --heading-font-family, --fixed-font-family

  /* Body sizes */
  --body-font-size-m, --body-font-size-s, --body-font-size-xs

  /* Heading sizes */
  --heading-font-size-xxl, --heading-font-size-xl, --heading-font-size-l
  --heading-font-size-m, --heading-font-size-s, --heading-font-size-xs

  /* Nav */
  --nav-height
}
```

## CSS Build Pipeline

The build uses **Gulp** with `postcss-import` to merge `_*.css` partials into compiled `*.css` outputs.

The pipeline:
1. Reads `brand-config.json` for the list of brands and themes
2. For each brand+theme combination, resolves the import chain
3. Generates compiled `styles.css`, `tokens.css`, `fonts.css`, `lazy-styles.css` in each directory
4. Also compiles block-level CSS for each brand+theme combination

**If builds fail:**
```sh
npm run scaffold:build:log   # Verbose output for debugging
```

Check for:
- Missing `@import` paths in `_*.css` partials
- Invalid CSS syntax
- Missing directories or files referenced in imports
- Memory: `NODE_OPTIONS='--max-old-space-size=4096'` is already set in the script

## Troubleshooting

**Styles not updating:**
- Ensure you edited the `_file.css` (underscore prefix), not the compiled `file.css`
- Run `npm run scaffold:build` after edits
- Hard-refresh the browser (Cmd+Shift+R)

**Build errors:**
- Run `npm run scaffold:build:log` for verbose error messages
- Check `@import` paths in your `_*.css` files are correct (relative paths)
- Verify the brand/theme name in `brand-config.json` matches the folder name exactly

**Brand/theme not loading in browser:**
- Confirm the brand is in `brand-config.json`
- Confirm compiled CSS exists (`file.css` without underscore)
- Check the AEM page metadata or URL path maps to the brand

**New brand has no block overrides:**
- The scaffold creates `_{block-name}.css` files for all existing blocks
- For blocks created after the brand was scaffolded, manually create brand override files or run `scaffold:create` for a new block (which reads `brand-config.json`)

## Definition of Done

The brand or theme is not complete until every item below passes. Do not mark the task done or move to the next step if any item is unresolved.

### New Brand
- [ ] `npm run scaffold:create` → Brand completed without errors
- [ ] `brand-config.json` updated with new brand name
- [ ] `styles/{brand}/_tokens.css` configured — **zero hardcoded values**, every value is a `var(--...)` reference
- [ ] All new token names follow W3C DTCG naming (`--{type}-{semantic}-{variant}`) — no brand/theme name embedded
- [ ] `styles/{brand}/_fonts.css` has `@font-face` declarations (if custom fonts used)
- [ ] Font files present in `styles/{brand}/fonts/` (if custom fonts used)
- [ ] `styles/{brand}/_styles.css` overrides reference tokens only — no raw literals or color keywords
- [ ] Block `_{block}.css` partials updated for all blocks with brand visual differences
- [ ] `npm run scaffold:build` ran without errors
- [ ] Dev server tested with `brand: {brand}` metadata on a page
- [ ] Brand tokens visible in DevTools `:root`
- [ ] Brand CSS files loading in Network tab
- [ ] `npm run lint` passes

### New Theme
- [ ] `npm run scaffold:create` → Theme completed without errors
- [ ] `brand-config.json` updated with new theme name
- [ ] `styles/themes/{theme}/_tokens.css` configured — **zero hardcoded values**, every value is a `var(--...)` reference
- [ ] All new token names follow W3C DTCG naming — no brand/theme name embedded
- [ ] `styles/themes/{theme}/_styles.css` overrides reference tokens only (if used)
- [ ] Brand+theme override files (`styles/{brand}/themes/{theme}/_tokens.css`) configured where needed
- [ ] Block theme partials updated for all blocks with theme visual differences
- [ ] `npm run scaffold:build` ran without errors
- [ ] Dev server tested with the theme active on a page
- [ ] Theme tokens visible in DevTools `:root`
- [ ] `npm run lint` passes
